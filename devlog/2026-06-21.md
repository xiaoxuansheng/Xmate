## V2.1.0 改动汇总

### A. Phase A Deleted 过滤 (file_trigram_index.dart)

**`readPostingsFiltered(raf, trigram, deletedIds)`** — 双指针线性过滤:
- postings delta-decode 后与 sorted deletedIds 做双指针合并
- 当命中删除时，先跳过所有连续重复 eid 再前进 deleted 指针（防御性，避免重复导致漏删）
- 返回 (filteredPostings, skippedCount)

**`readPostingsFromBytes(source, trigram)`** — 从内存字节读取 postings（delta 搜索用）

**`searchTrigrams(deletedIds:)`** — 新增可选参数，内部调用 readPostingsFiltered
- 累计日志: `[searchTrigrams] deleted filter: skipped N / M entries`

**`searchTrigramsMem(data, index, trigrams, {idOffset})`** — 纯内存 trigram 搜索（delta）
- 每个 entryId += idOffset（virtualId 映射）

**`buildToBytes(entries)`** — 在内存中构建 trigram 索引 → 返回 (index, raw bytes)

### B. Delta 文件 (.xmd)

**新文件: `lib/core/search/file_index_delta.dart`**

格式:
```
magic XFMD(4B) + version(2B) + flags(2B) + baseEntryCount(4B) + deletedCount(4B) + addedCount(4B)
+ deletedIds[deletedCount × 4B] (sorted ascending, absolute base entryId)
+ addedEntryTable[addedCount × 16B]
+ stringPoolLen(4B) + stringPool[N bytes]
+ nameTrigramIndex (trigramCount + entries)
+ pinyinTrigramIndex (same format)
+ CRC32(4B)
```

关键特性:
- 完全自包含（inline string pool + trigram indices）
- 原子写入（tmp → rename）
- CRC32 完整性校验
- `decodeEntry(deltaIdx)` — 从 inline pool 解码
- `buildDeltaBytes()` — 从 added entries + deleted IDs 组装完整 .xmd 字节

### C. LoadedSegment 支持 delta

**修改: `lib/core/search/file_index_store.dart`**

- `LoadedSegment.delta` — 新增 `FileIndexDelta?` 字段
- `loadSegment()` 末尾自动尝试 `FileIndexDelta.loadSync(xmdPath)` 并 attach

### D. 目录指纹 + 局部扫描

**修改: `lib/core/search/file_search_service.dart`**

**`_captureDirFingerprint(rootPath)`**:
- 递归 walk 所有子目录 → `Map<relDir, mtimeMs>` 
- 只 `statSync` 目录，不枚举文件 → O(目录数)

**`_findDirtyDirs(rootPath)`**:
- 对比 current vs snapshot → (dirtyDirs, deletedDirs)
- 日志: `[update] N dirs → X dirty, Y deleted`

**`_buildIndexedLookup(seg, delta)`**:
- 解码所有 base entries + delta added → `Map<lookupKey, entryId>`
- 排除 delta.deletedIds 中的 base entry

**`_lookupKey(path, name, ext)`** / **`_lookupKeyFromRelPath(relPath)`**:
- 统一的 lowercase key 格式: `"sub1/name.ext"` 或 `"name.ext"`

### E. manualUpdatePath() 增量更新

流程:
1. 加载 segment + 现有 delta
2. `_findDirtyDirs()` 定位脏目录
3. 无变化 → 跳过
4. 对每个 dirtyDir: 枚举直接子文件 → 与 lookup 比较 → added/deleted
5. 对每个 deletedDir: 该目录下所有已索引条目 → deleted
6. **Rule 1**: 所有 deletedSet eid < seg.entryCount（不包含 delta-added virtualId）
7. **Rule 2**: 合并现有 delta — 构造 oldDelMap → allScanKeys 匹配 → revive 重创建文件
8. 日志: `[update] final delta: +N added, -M deleted, revived=R`
9. `FileIndexDelta.buildDeltaBytes()` → `FileIndexDelta.writeSync()`
10. 更新指纹 → 重载 segment

### F. Rebuild 清除 delta

- `rebuildPath()` 末尾 `FileIndexDelta.deleteIfExists(xmdPath)`
- 重建后重新采集目录指纹

### G. search() delta 合并

Phase A:
- 搜索 base trigrams 时传入 `deletedIds`（过滤在 postings 解码阶段完成）
- 搜索 delta trigrams（内存，idOffset = seg.entryCount）
- 合并 base + delta matchCounts → preScores → TopK

Phase B:
- virtualId < seg.entryCount → seg.decodeEntry(vid)
- virtualId >= seg.entryCount → delta.decodeEntry(vid - seg.entryCount)
- 边界校验 + 越界日志

### H. Settings UI delta stats

- `SegmentInfo` 新增 `deltaAdded` / `deltaDeleted`
- Segment card 显示 `δ +N/-M`（橙色）
